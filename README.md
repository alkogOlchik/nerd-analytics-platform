# nerd-analytics-platform
 
# Работа с Git: ветки `dev` и `main`

Краткая логика для команды **Нёрд-аналитика**. Прочитайте один раз — дальше всё на автомате.

---

## Структура репозитория

```
nerd-analytics-platform/
├── backend/      # я
├── frontend/     # фронт
├── ml/           # ML
├── docker-compose.yml
└── .env.example  # шаблон переменных — заполняете локально у себя
```

Папки существуют всегда и видны всем. Ветки — это ваше личное рабочее пространство пока задача не готова.

---

## Роли веток

| Ветка | Назначение |
|-------|-----------|
| **`main`** | Стабильная версия. Сюда попадает только то, что точно работает. Прямых пушей сюда **нет**. |
| **`dev`** | Общая «песочница» интеграции. Сюда вливаете фичи и фиксы через Pull Request. |
| **`feat/...`** | Новая функциональность — ваше рабочее пространство. |
| **`fix/...`** | Исправление бага. |

---

## Именование веток

Создавайте ветку **от актуального `dev`**, называйте по задаче (не по имени участника):

```
feat/auth-service          ← бэк
feat/chat-widget           ← фронт
feat/ai-classifier         ← ml
fix/token-refresh-bug      ← любой
```

---

## Что делать (пошагово)

### 1. Обновить `dev` и создать ветку

```bash
git checkout dev
git pull origin dev
git checkout -b feat/моя-задача
```

### 2. Коммитить с понятными сообщениями (пж 🙏)

```bash
git add .
git commit -m "feat: добавлена авторизация через JWT"
git commit -m "fix: исправлена проверка дублирующихся email"
```

Префиксы: `feat:` — новое, `fix:` — баг, `chore:` — настройки/инфраструктура, `docs:` — документация.

### 3. Отправить ветку на удалённый репозиторий

```bash
git push -u origin feat/моя-задача
```

### 4. Открыть Pull Request в `dev` (не в `main`)

В описании PR укажите:
- что сделано
- как проверить (если есть API — как вызвать эндпоинты)
- есть ли зависимости от других веток

Ждёте ревью. Минимум один участник смотрит и одобряет.

### 5. После мержа — чистим за собой

```bash
git checkout dev
git pull origin dev
git branch -d feat/моя-задача        # удалить локально
```

Ветку на удалённом репозитории можно удалить прямо в интерфейсе GitHub/GitLab после мержа.

## Подветки (когда задача большая)

Если внутри фичи появляется крупная подзадача — создаёте подветку от родительской, не от `dev`:

```bash
git checkout feat/ticket-service
git checkout -b feat/ticket-service/sla-control

# ... делаете работу ...

git checkout feat/ticket-service
git merge feat/ticket-service/sla-control
git branch -d feat/ticket-service/sla-control
```

Когда вся фича готова — PR как обычно в `dev`.

---

## Как попадает код в `main`

```
feat/... → (PR) → dev → (PR) → main
```

В `main` мержим только когда фича протестирована в `dev` и команда договорилась. Это делаем вместе.

---

## Частые ситуации

**Кто-то запушил в `dev`, мне нужны его изменения:**
```bash
git checkout dev
git pull origin dev
git checkout feat/моя-задача
git merge dev       # или git rebase dev
```

**Накосячил и хочу откатить последний коммит (ещё не запушил):**
```bash
git reset --soft HEAD~1
```

**Случайно начал делать задачу прямо в `dev`:**
```bash
git stash
git checkout -b feat/моя-задача
git stash pop
```

## .env — важно

Файл `.env` с реальными секретами **никогда не коммитить**. Он уже в `.gitignore`.

Шаблон с нужными переменными лежит в `.env.example` — его заполняет каждый у себя локально. Если добавляете новую переменную окружения, обновляйте `.env.example` и пишите об этом в PR.
